# 個人ブログの技術スタックとして Astro + Svelte を採用する

Status: Accepted

# Context

個人ブログを新規に構築する。

本プロジェクトの目的は、技術記事を継続的に公開することに加え、新しい技術スタックの学習および実験の場とすることである。

ブログの主な要件は以下の通りである。

* 記事コンテンツを中心とした静的サイトであること
* 記事本文を Markdown で管理できること
* SEO および表示速度を重視すること
* 運用コストを低く抑えること
* ソースコードを公開可能であること
* 技術学習および実験の場として利用できること

想定しているインタラクティブ機能は限定的であり、主に以下を想定している。

* ダークモード切替
* 目次表示
* 検索 UI
* コードブロックのコピー

サイト全体を Single Page Application (SPA) として構築する必要はなく、可能な限りクライアントサイド JavaScript を削減したい。

また、サーバーやデータベースを運用せず、静的ホスティングサービス上で運用することを前提とする。

# Decision

以下の技術構成を採用する。

* Static Site Generator: Astro
* UI Framework: Svelte
* Styling: Panda CSS
* Content Management: Markdown + Astro Content Collections
* Hosting: GitHub Pages
* CI/CD: GitHub Actions

## Reason

### Astro + Svelte + Panda CSS を採用する理由

本プロジェクトはコンテンツ主体の静的サイトであり、ほとんどのページをビルド時に生成できる。そのため、静的サイト生成を前提としたアーキテクチャが適している。

Astro は静的サイト生成を前提としつつ、必要な箇所のみをクライアントサイドで動作させる Islands Architecture を採用している。また、Markdown および Content Collections によるコンテンツ管理機能を標準で提供しており、記事管理との親和性が高い。

一方、本プロジェクトではダークモード切替や検索 UI など、限定的なインタラクションを実装する可能性がある。Svelte は少ないコード量でインタラクティブな UI を実装できるため、このような小規模なクライアント機能との相性が良い。

Astro は必要なコンポーネントのみを Island としてハイドレーションする設計を採用しており、コンパイル時最適化を前提とする Svelte と組み合わせることで、静的サイトとしての性能を維持しながら必要な箇所のみをインタラクティブにできる。

スタイリングには Panda CSS を採用する。Panda CSS は型安全なスタイリングを提供し、デザイントークンを TypeScript 上で管理できる。また、ビルド時に静的 CSS を生成するため、ランタイムコストが発生しない。

本プロジェクトでは可能な限りクライアントサイド JavaScript を削減したいため、Astro、Svelte、Panda CSS の組み合わせによる Zero Runtime を志向したアーキテクチャを採用する。

また、本プロジェクトは技術学習も目的としているため、既存知識の延長ではなく、新しい UI フレームワークおよび CSS ソリューションを採用する価値があると判断した。

### GitHub Pages を採用する理由

本サイトは静的サイトであり、サーバーサイド処理を必要としない。

GitHub Pages を利用することで、インフラ運用コストを最小化し、コンテンツ作成および技術検証に注力できる。

また、GitHub Actions と組み合わせることで Git ベースのシンプルなデプロイパイプラインを構築できる。

### 検討した代替案

| Alternative   | 採用しなかった理由                                                                                                     |
| ------------- | ------------------------------------------------------------------------------------------------------------- |
| Next.js       | コンテンツ主体の静的サイトという要件に対して機能過多であると判断した。                                                                           |
| Gatsby        | React および GraphQL ベースのアーキテクチャを前提としており、Markdown 中心のシンプルな構成には複雑性が高いと判断した。また、新しい UI フレームワークを学習したいという目的とも一致しなかった。 |
| WordPress     | Git ベースでコンテンツを管理したい。また、データベースや管理画面、プラグイン更新などの運用コストは不要であると判断した。                                                |
| Astro Only    | 限定的なインタラクションを追加する可能性があること、および UI フレームワークの学習を目的としているため採用しなかった。                                                 |
| Astro + React | 既存知識の延長ではなく、新しい UI フレームワークを学習することを優先した。                                                                       |
| Astro + Vue   | 既存知識の延長ではなく、新しい UI フレームワークを学習することを優先した。                                                                       |
| Tailwind CSS  | Utility-first アプローチは強力であるが、本プロジェクトでは型安全性とセマンティックなスタイリングを重視した。また、デザイントークンを TypeScript 上で一元管理したいという要求から採用しなかった。 |
| CSS Modules   | シンプルな構成を実現できるが、デザイントークン管理や型安全性を重視し、より統一的なスタイリング基盤を提供する Panda CSS を採用した。                                       |
| SCSS          | 従来から利用実績のある手法であるが、型安全性およびデザイントークンの一元管理を重視したため採用しなかった。                                                         |

# Consequences

## Positive

* 高速な静的サイトを構築できる
* 運用コストを低く抑えられる
* Markdown ベースで記事を管理できる
* 必要な箇所だけクライアントサイド機能を追加できる
* GitHub 上で完結した開発・公開フローを構築できる
* 型安全なスタイリングを実現できる
* デザイントークンを TypeScript 上で一元管理できる
* クライアントサイド JavaScript を最小限に抑えられる
* Astro、Svelte、Panda CSS の学習機会を得られる

## Negative

* GitHub Pages の制約によりサーバーサイド機能を利用できない
* 将来的に認証や API が必要になった場合、ホスティング構成の見直しが必要になる
* Svelte および Panda CSS に関する知見が少ないため、初期の開発速度が低下する可能性がある
* SSG を前提としているため、リアルタイム性の高い機能との相性は良くない
* 一般的な React + Tailwind 構成と比較すると、コミュニティ規模やサンプルコードは限定的である

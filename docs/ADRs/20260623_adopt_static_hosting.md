# 静的ホスティングアーキテクチャを採用する

* Status: accepted
* Date: 2026-06-23

## コンテキストと問題提起

個人ブログを新規に構築するにあたり、インフラ構成の基本方針を決める必要がある。個人が継続的に運用することを前提とするため、金銭的コストと管理工数の最小化が重要な関心事となる。

**この決定が答える問い：** 個人運用において持続可能な、低コスト・低メンテナンスかつクレジットカード登録不要なインフラ構成はどれか？クレジットカードを登録しないことで、設定ミスや想定外の利用による請求リスクをゼロにしたい。

## 検討した選択肢

* GitHub Pages
* Cloudflare Pages
* Vercel / Netlify

## 決定の結果

選択肢：**GitHub Pages**、なぜなら現時点でサーバーサイド処理を必要とせず、追加のアカウント作成なしで運用を開始できる最もシンプルな選択肢だからである。GitHub Pages は既存の GitHub アカウントと統合されており、GitHub Actions との組み合わせで Git ベースのデプロイパイプラインを構築できる。Cloudflare Pages はリポジトリとの連携が手厚くプライベートリポジトリにも対応するが、新規アカウント作成が必要でその工数に見合うメリットが現時点では薄い。Vercel / Netlify はサーバーサイド機能も持てるが、現時点では不要な機能に対してコストが発生し、また無料枠を超えた際の請求リスクを避けるためクレジットカード登録を要しない構成を優先した。

### 結果・影響

* Good: サーバー・DB 不要のため月額コストがゼロ
* Good: GitHub Actions との統合によりシンプルなデプロイパイプラインを構築できる
* Good: インフラ管理の工数がかからず、コンテンツ作成に集中できる
* Bad: サーバーサイド処理・認証・動的 API が必要になった場合、ホスティング構成の見直しが必要になる
* Bad: GitHub Pages にはカスタムドメインや証明書まわりに制約がある
